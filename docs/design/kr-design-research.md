# Kingdom Rush 及同类塔防游戏 Design 调研报告

> 作者：Vivian（Designer）
> 日期：2026-04-07
> 目标：为 Tower Storm Phase 2 的视觉升级、UI 重构、素材策略提供依据

---

## 1. 结论先行

### 我对 Tower Storm 的设计建议

**产品形态：** 对齐 Kingdom Rush 经典塔防体验，不做每 wave 重置的 puzzle 玩法。  
**视觉路线：** 采用 **高清卡通手绘风**，但先从 **“轻量手绘 + 可快速落地”** 开始，不一步追求 KR 级细节密度。  
**素材方案：** 采用 **免费资源包为主 + AI 生成补充 + 统一后处理** 的混合方案。  
**UI 路线：** 优先重构 **建塔交互**，从列表/面板式改为 **环形建塔菜单**；其次优化 HUD 图标化与反馈动画。  
**Phase 2 优先级：**
1. 环形建塔菜单
2. 地图 tileset 替换纯色块
3. 敌人/塔从几何图形升级为 sprite
4. 建塔/攻击/死亡基础动画
5. HUD 图标化与信息层级优化

## 2. Kingdom Rush 的设计特征拆解

## 2.1 视觉风格

Kingdom Rush 的核心不是“写实”，而是：
- **高辨识度卡通手绘**
- **清晰轮廓 + 明确剪影**
- **背景相对克制，敌人与塔更亮更突出**
- **轻喜剧、高幻想氛围**

### 关键设计原则
- **敌人优先可读**：敌人拥有更极端的明暗对比，保证战斗中一眼识别
- **地图服务玩法，不抢戏**：环境细节丰富，但不会压过敌人/塔/UI
- **每类塔有强记忆点**：弓箭塔、兵营、魔法塔、炮塔在造型、颜色、功能上都一眼区分
- **“手绘感”来自细节而非复杂度**：轮廓线、阴影块面、局部纹理、趣味动画共同构成风格

### 对我们的启发
Tower Storm 当前的问题不是“颜色不好看”，而是：
- 图形还是 placeholder 几何语言
- 缺乏轮廓和材质信息
- UI 仍偏工具化，不像完整游戏

所以 Phase 2 重点不是继续微调纯色，而是要把 **图形语言从“原型”切换到“游戏资产”**。

---

## 2.2 UI 系统

### Kingdom Rush 的 UI 优点
- **HUD 极简**：只保留金币、生命、波次等核心信息
- **建塔点可视化清晰**：空建塔点天然可点击、无学习成本
- **环形建塔菜单**：点击建塔点后，4 个塔选项围绕中心展开
- **对象就地操作**：操作围绕建塔点或塔本体发生，而不是把用户视线拉去屏幕边缘
- **移动端友好**：圆形按钮、大触区、等距分布，适合手指点击

### 对我们的启发
当前 Tower Storm 的建塔菜单已经比最初版本好很多，但依然是 **小面板 + 文本信息**，更像开发工具，不像成品游戏。

**建议：Phase 2 重构为 KR 风格环形菜单**
- 中心：建塔点
- 上 / 右 / 下 / 左：4 个塔类型
- 每个选项包含：图标 / 成本 / 可建状态
- 买不起置灰
- 点击空白处关闭
- 展开动画：从中心“弹出”

这是最值得优先做的体验升级点。

## 2.3 动画体系

KR 的高级感，很大一部分来自 **动画反馈密度**。

### 关键动画节点
- 建塔：放置出现 + 轻微弹跳
- 攻击：塔体开火 / 抖动 / 闪光
- 敌人死亡：缩小 / 淡出 / 爆炸 / 被击退
- 波次开始：视觉提示、按钮状态变化
- 菜单展开：环形选项从中心生长出来
- 环境细节：风车、旗帜、草地微动

### 对我们的建议
Phase 2 不需要一口气做满 KR 的动画复杂度，但至少应补齐这 4 个：
1. 建塔弹跳动画
2. 炮塔命中 AoE 扩散动画
3. 敌人死亡淡出/缩放
4. 环形菜单展开动画

做到这一步，体验就会从“可玩 demo”明显跃迁到“游戏产品”。

## 3. 同类游戏对比

## 3.1 Bloons TD 6

### 优点
- 角色辨识度极强
- UI 信息密度高但组织有序
- 升级和数值表达非常直接

### 缺点
- 偏移动端感，PC 上有时显得 UI 太重
- 视觉风格更偏低模卡通，不适合直接作为我们主风格参考

### 对我们的启发
- 可以学习它的 **敌我辨识度** 和 **升级信息表达**
- 不建议学习它的整体美术风格

## 3.2 Plants vs Zombies

### 优点
- 极致清晰的可读性
- 图标和角色设计非常强
- 每个单位的功能和视觉天然绑定

### 对我们的启发
- 可以借鉴它的 **功能视觉化能力**：一眼知道这个单位是快、重、远程还是近战

## 4. 我们与 KR 的差距

## 4.1 美术层
- 现在：几何图形 + 色块占位
- KR：完整 sprite + 阴影 + 轮廓 + 趣味细节

## 4.2 UI 层
- 现在：HUD 可用，但仍偏基础；建塔菜单偏面板式
- KR：对象就地操作 + 环形选择 + 更强图标化

## 4.3 动画层
- 现在：局部 tween 已有，但整体反馈密度不足
- KR：战斗、建塔、环境、菜单都有动态反馈

## 4.4 世界观层
- 现在：有塔防结构，但还没有“世界”感
- KR：地图本身就有故事和个性

## 5. Tower Storm 的设计升级路线

## Phase 2（优先做）
### A. UI 重构
- 建塔菜单 → 环形菜单
- HUD 文字 → 图标 + 数字
- 波次倒计时可视化

### B. 美术升级
- 地图：引入 tileset（草地、路径、障碍物、装饰）
- 敌人：替换为基础 sprite
- 塔：替换为基础 sprite

### C. 动画补齐
- 建塔弹跳
- 敌人死亡
- 炮塔爆炸
- 菜单展开

## Phase 3-4（进一步打磨）
- 地图环境装饰（风车、旗帜、植被）
- 更细的攻击特效
- UI 皮肤统一
- 结算界面做成成品态

## 6. 素材来源方案

## 推荐方案：免费资源包为主 + AI 生成补充

### 为什么推荐这个方案
**只用 AI：** 风格一致性差，后处理成本高  
**只用免费素材：** 风格容易拼不起来，缺少关键塔或敌人  
**混合方案：** 速度、成本、统一性三者最平衡

### 具体执行方式
1. 先选一套主资源风格，作为视觉基线
2. 不足的塔、敌人、BOSS、UI 插图再用 AI 补
3. 我统一做后处理：
   - 调色
   - 描边
   - 阴影一致化
   - 尺寸裁切
   - 命名和导出规范化

## 7. 当前素材筛选建议

我初步看下来，**直接找到“免费 + 非像素 + 接近 KR 手绘卡通风”的完整塔防包并不容易**。大多数免费的 tower defense 资源更偏：
- 像素风
- 极简 low-fi
- 塔防功能素材但缺少完整 UI

所以我建议素材筛选策略不要只找“完整包”，而是拆成 3 类：

### A. 地图 / tileset
目标：找卡通地面、路径、树木、石头、建筑基底

### B. 塔 / 敌人 sprite
目标：至少先有“弓箭塔 / 炮塔 / 普通兵 / 快速兵”可用版本

### C. UI 元素
目标：按钮、面板、图标边框、徽章

这样组合起来，反而更容易凑出统一效果。

## 8. 对开发的明确设计 ask

<@1488911832340234421> Haaland 后续实现时，我建议你优先支持这些结构：

1. **环形菜单容器化**
- 允许 4 个方向等距摆放按钮
- 可扩展到升级/出售二级菜单

2. **Sprite 替换能力**
- 塔 / 敌人从 Graphics 切换为 sprite 时，保留当前位置、碰撞逻辑、range 逻辑

3. **动效接口预留**
- buildTower 时支持 playBuildAnimation()
- enemy death 时支持 playDeathEffect()
- cannon hit 时支持 playImpactEffect()
