# Tower Storm — Style Guide（Phase 2 初稿）

> 作者：Vivian（Designer）
> 日期：2026-04-07
> 状态：Phase 2 可落地初稿

---

## 1. 目标

本指南用于把 Tower Storm 从 placeholder 原型推进到更接近 Kingdom Rush 的成品体验。

本稿重点覆盖：
- 视觉风格基线
- 4 塔 / 4 敌人的区分规则
- 环形建塔菜单规范
- 动画与反馈规范
- 移动端基础兼容预案
- 面向开发的可落地参数

---

## 2. 风格关键词

- 高清卡通
- 轻手绘感
- 高辨识度
- 背景克制、战斗元素突出
- 好用优先于好看

**一句话原则：**
先保证玩家 0.5 秒内看懂“这是什么、危险吗、该怎么应对”，再谈装饰。

---

## 3. 视觉层级

### 3.1 优先级
1. 敌人
2. 塔 / 士兵 / 弹道
3. 关键 HUD 信息
4. 建塔点 / 交互菜单
5. 地图装饰

### 3.2 执行规则
- 敌人和塔使用更高对比度、更清晰轮廓
- 地图颜色饱和度略低于战斗元素
- HUD 用深色底托住文字，不直接压在地图上
- 同类单位优先用**形状 + 颜色**双重区分，不只靠颜色

## 5. 形状语言规范

## 5.1 四种塔

### 弓箭塔 Archer
- 关键词：轻、稳、基础、远程
- 轮廓：圆形塔头 + 方形底座
- 识别特征：顶部有箭头/弓形指示
- 体量：中等
- 形状建议：**圆 + 小尖头**

### 魔法塔 Magic
- 关键词：高、尖、能量感、神秘
- 轮廓：比弓箭塔更高、更尖
- 识别特征：顶部能量球/水晶
- 体量：偏瘦高
- 形状建议：**尖塔 + 发光核心**
- 开发建议：后续 sprite 替换前，placeholder 也尽量表现“上窄下宽”的轮廓，不要只做圆球

### 炮塔 Cannon
- 关键词：厚重、爆发、机械感
- 轮廓：宽底座 + 短粗炮管
- 识别特征：炮口朝向明显
- 体量：最厚重
- 形状建议：**矩形底 + 短炮管**

### 兵营 Barracks
- 关键词：驻守、群体、近战、组织感
- 轮廓：建筑感强于“武器感”
- 识别特征：旗帜 / 盾牌 / 门
- 体量：偏宽，不强调高度
- 形状建议：**小房屋 / 营地 / 盾牌图形**
- 状态要求：需要明显区分“可出兵 / 冷却中 / 士兵已阵亡待重生”

## 5.2 四种敌人

### 普通兵 Normal
- 关键词：标准、基础、无特性
- 轮廓：最简单
- 形状建议：**圆角矩形 / 小步兵**

### 快速兵 Fast
- 关键词：轻、快、脆
- 轮廓：更小、更尖
- 形状建议：**菱形 / 前倾三角感**
- 目的：即使不看颜色，也能感觉它“跑得快” 

### 重甲兵 Heavy
- 关键词：宽、重、慢、钝
- 轮廓：比普通兵更宽更厚
- 形状建议：**宽矩形 + 肩甲块感**
- 识别特征：大肩甲 / 厚盾 / 低重心

### 飞行单位 Flying
- 关键词：轻、飘、空中、绕过地面
- 轮廓：横向展开
- 形状建议：**带翼轮廓 / 菱形主体 + 双翼**
- 识别特征：翅膀、阴影、与地面单位的高度感差异
- 最低要求：必须和地面单位在轮廓上强区分，避免只靠颜色判断

## 6. 尺寸与比例

### 6.1 塔
- 基础占位：64 × 64
- 可绘制主体：56 × 56
- 底座建议：48 × 48

### 6.2 敌人
- 普通 / 快速：20-24 px 主体
- 重甲：28 px 主体
- 飞行：24-28 px 主体，翼展大于本体

### 6.3 士兵（兵营单位）
- 主体：16-20 px
- 与敌人相比更小，但不能小到看不清交战关系
- 需要单独血条或至少受击反馈

## 7. 环形建塔菜单规范

## 7.1 目标
相比当前横向面板，环形菜单更接近 KR，且更适合触控。

## 7.2 结构
- 中心：建塔点
- 上：弓箭塔
- 右：魔法塔
- 下：炮塔
- 左：兵营

> 实际方向可按代码实现微调，但必须保持固定肌肉记忆，不要每次变位置。

## 7.3 组件元素
每个塔按钮包含：
- 圆形底
- 塔图标
- 价格
- 可建 / 不可建状态

## 7.4 状态
- 可建：100% 不透明 + 彩色
- 买不起：40% 不透明 + 灰化
- hover / focus：放大 1.08x + 外圈高亮
- 点击：缩放回弹

## 7.5 动画
- 展开：从中心向外弹出
- 时长：180-220ms
- easing：`Back.Out` 或类似轻弹感曲线

---

## 8. HUD 规范

### 8.1 桌面端
- 顶部固定深色背景条
- 顺序：生命 > 金币 > 波次 > 剩余敌人 > 波次按钮
- 文案尽量图标化，不用长句

### 8.2 移动端基础兼容（Phase 2）
目标：**能看、能点、不错位**，不追求完整移动体验

- 横屏优先
- 触控热区：**≥ 48px**
- HUD 字号较桌面放大 10-15%
- 波次按钮和建塔点优先保证可点击
- 若竖屏，允许先给旋转提示，不强行完整支持

### 8.3 移动端完整适配（Phase 4）
- HUD 重排
- 建塔菜单移动端专用交互
- 安全区 / 刘海屏处理
- 横竖屏策略明确化

## 10. 兵营专项规范

这是最容易做得“逻辑对了但体验不清楚”的塔，必须写清楚。

### 10.1 状态
- 空闲：可出兵
- 交战中：士兵已拉出去
- 士兵死亡：重生倒计时
- 升级中 / 已升级：视觉层级变化

### 10.2 视觉要求
- 兵营本体要有“建筑”感
- 士兵必须与敌人明显区分
- 交战时要能看出“谁在打谁”
- 冷却中建议有简单倒计时或颜色状态变化

### 10.3 开发对齐点
Haaland 实现时至少要支持：
- 士兵与兵营分离渲染
- 兵营状态可视化
- 飞行单位跳过士兵判定

## 11. 面向开发的直接参数建议

### 11.1 魔法塔
- 主色：`0x7C4DFF`
- 高亮：`0xB388FF`
- 轮廓：上尖下稳
- 图标关键词：水晶 / 能量球

### 11.2 重甲兵
- 主色：`0x78909C`
- 形状：更宽、更厚
- 图标关键词：肩甲 / 盾 / 厚甲板

### 11.3 飞行单位
- 主色：`0xCE93D8`
- 必须增加双翼或悬浮特征
- 建议附带轻微阴影，表达离地感

### 11.4 兵营
- 主色：`0xFFB300`
- 辅助元素：旗帜 / 盾牌 / 木门
- 士兵色彩不要完全同兵营，避免糊成一团

## 13. 结论

Phase 2 的设计重点不是继续修饰 placeholder，而是完成这三件事：
1. 建立 4 塔 / 4 敌人的清晰形状语言
2. 把建塔交互逐步推向 KR 的环形菜单体验
3. 为后续 sprite 替换和移动端兼容预留统一规范

如果执行到位，Tower Storm 会从“能玩的原型”明显进入“有产品感的塔防游戏”。
